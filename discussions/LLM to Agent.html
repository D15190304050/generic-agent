① Workflow / DAG Agent（工程派 / 控制派）

本质

常见实现

  • State Machine

优点（工程上非常重要）

✅ 可控性极强
✅ 可调试、可测试、可回放
✅ 失败模式清晰

❌ 不灵活

这是“LLM-enhanced Workflow”，不是“LLM-native Agent”

② ReAct Agent（推理派 / 研究派）

本质

Agent = LLM 在一个 loop 里自我驱动

✅ 泛化能力强
✅ prompt 一写就能跑
✅ demo 非常好看

缺点（工程灾难区）

❌ 不可控
❌ 不可预测
❌ 成本和延迟不稳定
❌ Debug 极其痛苦
❌ 容易无限循环 / hallucination

本质评价

这是“LLM-as-Controller”

适合：

  • 研究

  • 原型

  • 小工具

  • 非关键路径

✅ 可组合性强

✅ 非常适合复杂系统

五、核心接口设计（重点）

1️⃣ Agent（最核心）

java
public interface Agent {

    AgentId id();

    AgentResult handle(AgentContext context, AgentRuntime runtime);

}

设计含义：

  • Agent 不持有全局状态

  • 一次 handle = 一次“控制权持有期”

  • runtime 用来：

    • 请求 handoff

    • 查询系统能力

    • 访问 registry

2️⃣ AgentContext（不是 Chat History）

java
public interface AgentContext {

    ConversationId conversationId();

    Map<String, Object> state();

    void put(String key, Object value);

    <T> T get(String key, Class<T> type);
}

关键思想：

Context 是“共享工作记忆”，不是 prompt

  • LLM prompt 只是 context 的一个投影

  • Context 是跨 Agent 传递的

3️⃣ AgentResult（控制权声明）

java
public sealed interface AgentResult
        permits Continue, Finish, HandoffResult {
}

java
public final class Continue implements AgentResult {
}

java
public final class Finish implements AgentResult {
}

java
public final class HandoffResult implements AgentResult {

    private final AgentId target;
    private final String reason;

}

👉 非常重要：
Agent 不直接调用别的 Agent，只能“声明意图”。

4️⃣ AgentRuntime（系统中枢）

java
public interface AgentRuntime {

    void handoff(AgentId target, AgentContext context);

    Optional<Agent> findAgentByCapability(String capability);

    LlmClient llm();

}

含义：

  • Runtime = orchestration

  • Agent = decision unit

六、三种 Agent 如何实现（关键解惑）

① WorkflowAgent（DAG）

java
public class WorkflowAgent implements Agent {

    private final Workflow workflow;

    @Override
    public AgentResult handle(AgentContext context, AgentRuntime runtime) {
        WorkflowNode node = workflow.next(context);
        node.execute(context);
        return workflow.isFinished(context)
                ? new Finish()
                : new Continue();
    }
}

解释：

  • DAG 是 Agent 内部实现

  • 外部只看到一个 Agent

👉 Workflow ≠ Agent
👉 Workflow ∈ Agent implementation

② ReactiveAgent（ReAct）

java
public class ReactiveAgent implements Agent {

    @Override
    public AgentResult handle(AgentContext context, AgentRuntime runtime) {
        String thought = runtime.llm().complete("What should I do?");
        if (thought.contains("search")) {
            // tool call
        }
        return new Continue();
    }
}

解释：

  • ReAct 是一种 decision loop

  • 不影响 Agent 抽象

③ DelegatingAgent（Agent + Handoff）

java
public class RouterAgent implements Agent {

    @Override
    public AgentResult handle(AgentContext context, AgentRuntime runtime) {
        String intent = runtime.llm().complete("Classify user intent");

        AgentId target = switch (intent) {
            case "search" -> new AgentId("search-agent");
            case "analysis" -> new AgentId("analysis-agent");
            default -> new AgentId("default-agent");
        };

        return new HandoffResult(target, "intent-based routing");
    }
}

解释：

  • Agent 的核心职责：决定“我是不是该继续”

  • Handoff 是一等公民

而是没有把“实现细节”误提升为“一等抽象” ——这是好事。

二、MCP 到底是什么？为什么你“找不到它”

我们先把 MCP 去魅一下。

1️⃣ MCP 的本质（工程视角）

MCP ≠ Agent
MCP ≠ RAG
MCP = LLM ↔ 外部能力 的通信协议层

换句话说：

  • MCP 解决的是：

    • tool schema

    • tool discovery

    • tool invocation

    • tool transport（stdio / http / ws）

  • 它不关心 Agent 的控制权

  • 它不关心 workflow / handoff

👉 MCP 是“IO 层协议”

2️⃣ MCP 在你现在架构里的“精确位置”

看你的结构：

text
agent
runtime
llm
adapters

MCP 的归属非常明确：

MCP = llm + adapters 这一层里的实现方式之一

更具体一点：

text
llm
├── ILlmClient
└── IToolExecutor   ← 👈 MCP 在这里

text
adapters
└── mcp
    ├── McpToolExecutor
    └── McpTransport

为什么 MCP 不该出现在 core / agent？

因为：

  • Agent 只需要知道：“我能不能用工具”

  • MCP 决定的是：“工具是怎么连出去的”

这是非常标准的 Hexagonal Architecture（端口-适配器）。

3️⃣ 一个非常关键的判断（你要记住）

MCP 是“可替换的 Tool Backend”，不是 Agent 架构的一部分

今天你可以是：

  • MCP

  • 明天可以是：

    • HTTP API

    • gRPC

    • 本地 Java 方法

👉 如果 MCP 被设计进 Agent Core，你的 SDK 就死了

三、RAG：你“没看到”，但它其实已经在了

你说得非常敏锐：

“RAG 我也没看到，但我感觉 RAG 可能也是有的”

你的感觉是 100% 对的。

1️⃣ RAG 在工程上到底是什么？

我们用一句话定义：

RAG = 一种 Agent 的决策/执行策略，而不是基础设施

拆开来看：

  • Retrieval：查数据

  • Augmentation：放进上下文

  • Generation：交给 LLM

👉 没有任何一步要求“RAG 成为核心抽象”

2️⃣ RAG 在你当前设计中的“精确落点”

RAG 不在 core
不在 runtime
不在 llm

它在这两处之一（或同时）：

public class RagAgent implements IAgent {

    public AgentResult handle(AgentContext ctx, IAgentRuntime rt) {
        List<Document> docs = retriever.search(ctx.get("query"));
        ctx.put("retrieved_docs", docs);

        String answer = rt.llm().complete(
            buildPrompt(ctx, docs)

        return new Finish();
    }

👉 不需要额外“RAG 框架抽象”

3️⃣ 为什么你“感觉缺了 RAG”？

因为现在市面上的很多框架：

  • 把 RAG：

    • 做成 pipeline

    • 做成 first-class concept

  • 甚至：

    • RAG == Agent

但这是产品层决策，不是架构必然性。

四、把 MCP / RAG / Agent 放在同一张认知地图里
